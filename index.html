    2.1 Operations on Disposable Objects

    See Common Resource Management Interfaces (5.2.1).

    2.1.1 DisposableResource Records

    A DisposableResource is a Record value used to encapsulate a disposable object along with the method used to dispose the object. DisposableResource Records are produced by the CreateDisposableResource abstract operation.

    DisposableResource Records have the fields listed in Table 2:

    2.1.2 CreateDisposableResource ( V, hint )

    The abstract operation CreateDisposableResource with arguments V and hint performs the following steps:

    1. Assert: V is an ECMAScript language value.
    2. Assert: hint is either sync or async.
    3. Let disposeMethod be ? GetDisposeMethod(V, hint).
    4. Return the DisposableResource { [[ResourceValue]]: V, [[DisposeMethod]]: disposeMethod }.

    2.1.3 GetDisposeMethod ( V, hint )

    The abstract operation GetDisposeMethod with arguments V and hint performs the following steps:

    1. Assert: V is an ECMAScript language value.
    2. Assert: hint is either sync or async.
    3. If V is null or undefined, return undefined.
    4. If hint is async, then
      1. Let method be ? GetMethod(V, @@asyncDispose).
      2. If method is undefined, then
        1. Set method to ? GetMethod(V, @@dispose).
    5. Else,
      1. Let method be ? GetMethod(V, @@dispose).
    6. If method is undefined, throw a TypeError exception.
    7. Return method.

    2.1.4 Dispose ( V, hint, method )

    The abstract operation Dispose with arguments V, hint, and method performs the following steps:

    1. Assert: V is an ECMAScript language value.
    2. Assert: hint is either sync or async.
    3. Assert: method is a function object.
    4. If V is null or undefined, return undefined.
    5. Let result be ? Call(method, V).
    6. If hint is async and result is not undefined, then
      1. Perform ? Await(result).
    7. Return undefined.

    3.1.1 Environment Records

    3.1.1.1 Try-Using Environment Records

    A try-using Environment Record is a declarative Environment Record that is used to represent the scope of a try-using statement and to track resources that must be disposed.

    Try-Using Environment Records have the additional state fields listed in Table 3.

    Try-Using Environment Records support all of the declarative Environment Record methods listed in Table 16 and share the same specifications for all of those methods except for InitializeBinding. In addition, try-using Environment Records support the methods listed in Table 4:

    The behaviour of the additional concrete specification methods for try-using Environment Records is defined by the following algorithms:

    3.1.1.1.1 InitializeBinding ( N, V )

    1. Let envRec be the try-using Environment Record for which the method was invoked.
    2. Assert: envRec must have an uninitialized binding for N.
    3. Perform ? envRec.AddDisposableResource(V).
    4. Set the bound value for N in envRec to V.
    5. Record that the binding for N in envRec has been initialized.
    6. Return NormalCompletion(empty).

    3.1.1.1.2 AddDisposableResource ( V )

    1. Let envRec be the try-using Environment Record for which the method was invoked.
    2. If V is null or undefined, return NormalCompletion(empty).
    3. Let hint be envRec.[[Hint]].
    4. Let resource be ? CreateDisposableResource(V, hint).
    5. Append resource to envRec.[[ResourceStack]].
    6. Return NormalCompletion(empty).

    3.1.2 Lexical Environment Operations

    3.1.2.1 NewTryUsingEnvironment ( E, hint )

    When the abstract operation NewDeclarativeEnvironment is called with a Lexical Environment as argument E and argument hint the following steps are performed:

    1. Assert: hint is either sync or async.
    2. Let env be a new Lexical Environment.
    3. Let envRec be a new try-using Environment Record containing no bindings.
    4. Set envRec.[[Hint]] to hint.
    5. Set envRec.[[ResourceStack]] to a new empty List.
    6. Set env's EnvironmentRecord to envRec.
    7. Set the outer lexical environment reference of env to E.
    8. Return env.

    4.1.10 Runtime Semantics: TryUsingBodyEvaluation (Head, boundNames, Block, hint)

    The abstract operation TryUsingBodyEvaluation with arguments Head, boundNames, Block, and hint performs the following steps:

    1. Assert: Head is either an Expression or TryUsingDeclaration.
    2. Assert: boundNames is a List of String values.
    3. Assert: Block is a Block.
    4. Assert: hint is either sync or async.
    5. Let oldEnv be the running execution context's LexicalEnvironment.
    6. Let tryEnv be ! NewTryUsingEnvironment(oldEnv, hint).
    7. Let envRec be tryEnv's EnvironmentRecord.
    8. For each boundNames of boundNames, do
      1. Perform ! envRec.CreateImmutableBinding(boundName, true).
    9. Let completion be the result of evaluating Head.
    10. If completion.[[Type]] is normal, then
      1. If Head is an Expression, then
        1. Set completion to envRec.AddDisposableResource(completion.[[Value]]).
      2. Else, Head is a TryUsingDeclaration parse node and disposable resources were added when bindings were initialized.
    11. If completion.[[Type]] is normal, then
      1. Let completion be the result of evaluating Block.
    12. Let errors be a new empty List.
    13. If completion.[[Type]] is throw, then
      1. Append completion.[[Value]] to errors.
    14. For each resource of envRec.[[ResourceStack]], in reverse list order, do
      1. Let result be Dispose(resource.[[ResourceValue]], hint, resource.[[DisposeMethod]]).
      2. If result.[[Type]] is throw, then
        1. Append result.[[Value]] to errors.
    15. Set the running execution context's LexicalEnvironment to oldEnv.
    16. Let errorsLength be the number of elements in errors.
    17. If errorsLength > 1, then
      1. Let error be a newly created AggregateError object.
      2. Perform ! DefinePropertyOrThrow(error, "errors", PropertyDescriptor { [[Configurable]]: true, [[Enumerable]]: false, [[Writable]]: true, [[Value]]: ! CreateArrayFromList(errors) }).
      3. Return ThrowCompletion(error).
    18. If errorsLength = 1, then
      1. Let error be the first element of errors.
      2. Return ThrowCompletion(error).
    19. Return completion.

    4.1.11 Runtime Semantics: CatchFinallyEvaluation (tryCompletion, Catch, Finally)

    The abstract operation CatchFinallyEvaluation with arguments tryCompletion, Catch, and Finally performs the following steps:

    1. Assert: tryCompletion is a Completion Record.
    2. Assert: Catch is either a Catch or is undefined.
    3. Assert: Finally is either a Finally or is undefined.
    4. If tryCompletion.[[Type]] is throw and Catch is not undefined, let C be CatchClauseEvaluation of Catch with argument tryCompletion.[[Value]].
    5. Else, let C be tryCompletion.
    6. If Finally is not undefined, let F be the result of evaluating Finally.
    7. Else, let F be C.
    8. If F.[[Type]] is normal, set F to C.
    9. Return Completion(UpdateEmpty(F, undefined)).

    5 Control Abstraction Objects

    5.1 Iteration

    5.1.1 The %IteratorPrototype% Object

    5.1.1.1 %IteratorPrototype% [ @@dispose ] ( )

    The following steps are taken:

    1. Let O be the this value.
    2. Let return be ? GetMethod(O, "return").
    3. If return is not undefined, then
      1. Perform ? Call(return, O, « »).
    4. Return NormalCompletion(empty).

    5.1.2 The %AsyncIteratorPrototype% Object

    5.1.2.1 %AsyncIteratorPrototype% [ @@asyncDispose ] ( )

    The following steps are taken:

    1. Let O be the this value.
    2. Let promiseCapability be ! NewPromiseCapability(%Promise%).
    3. Let return be GetMethod(O, "return").
    4. IfAbruptRejectPromise(return, promiseCapability).
    5. If return is undefined, then
      1. Perform ! Call(promiseCapability.[[Resolve]], undefined, « undefined »).
    6. Else,
      1. Let result be Call(return, O, « undefined »).
      2. IfAbruptRejectPromise(result, promiseCapability).
    7. Return promiseCapability.[[Promise]].

Any object that provides all the properties as described by an interface's specification conforms to that interface. An interface is not represented by a distinct object. There may be many separately implemented objects that conform to any interface. An individual object may conform to multiple interfaces.
